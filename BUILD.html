1. What we are building

An analyst asks a question in English; the system writes SQL, runs it against a database, checks the result, and retries if it looks wrong. The model itself (Qwen3-30B) is served by vLLM; a LangGraph agent wraps it with a verify/revise loop; two observability stacks watch the machine (Prometheus + Grafana) and the logic (Langfuse).

5. Docker & docker-compose — concepts + how to read

There is no Dockerfile for the o11y stack — docker-compose.yml pulls pre-built public images and wires them together. The mental model:

Imagefrozen, read-only app template (like a class)
Containera running instance of an image (like an object), disposable
Volumedisk that survives restarts — keeps your Grafana/Langfuse data
Port map"3000:3000" = host_port:container_port
Networkcompose puts services on one private net; they reach each other by name (postgres:5432)

Reading a service block:

services:
  grafana:                         # service name = hostname on the private net
    image: grafana/grafana:11.3.0  # which prebuilt image + pinned version
    ports:
      - "3000:3000"                # VM:3000 -> container:3000  (you SSH-forward 3000)
    volumes:
      - ./infra/grafana/provisioning:/etc/grafana/provisioning:ro  # host:container:read-only
      - grafana-data:/var/lib/grafana                              # named volume, persists
    depends_on:
      - prometheus                 # start prometheus first

Three patterns to recognize:

  • depends_on + healthcheck — Langfuse waits for Postgres/ClickHouse/Redis/MinIO to be actually ready (e.g. pg_isready), which is why first boot is slow.
  • extra_hosts: host.docker.internal:host-gateway on Prometheus — the bridge from inside a container to the host. vLLM runs on the host, so Prometheus scrapes host.docker.internal:8000. That one line is why scraping works.
  • &langfuse-env anchor — a YAML reuse trick: define ~25 env vars once, splat into two services with <<: *langfuse-env.

6. The agent Dockerfile (optional)

The assignment never asks for this — on the VM you just run uvicorn. We added one to show the structure. Key idea: each line is a cached layer; order steps so editing code doesn't reinstall packages.

FROM python:3.12-slim         # 1. small official base
WORKDIR /app                  # 2. where code lives in the container
COPY requirements-dev.txt .   # 3. deps manifest FIRST (rarely changes)
RUN pip install -r requirements-dev.txt   #    cached unless the manifest changes
COPY agent/ ./agent/          # 4. code LAST (changes often)
EXPOSE 8001                   # 5. document the port
CMD ["uvicorn","agent.server:app","--host","0.0.0.0","--port","8001"]  # 6. the process
It installs only agent deps — never vllm. vLLM stays on the host GPU; the container reaches it via host.docker.internal:8000.

7. How to read the code we wrote

Every LLM node follows the same shape as the provided generate_sql_node — learn it once. LangGraph nodes don't mutate state; they return a dict of changed fields and the framework merges it.

def verify_node(state):
    result_view = state.execution.render() if state.execution else "NO EXECUTION RESULT"
    response = llm().invoke([
        ("system", prompts.VERIFY_SYSTEM),                # who you are + rules
        ("user",   prompts.VERIFY_USER.format(            # the task, data filled in
            question=state.question, sql=state.sql, result=result_view)),
    ])
    verdict = _parse_json_object(response.content)        # tolerate prose/fences
    ok    = bool(verdict.get("ok", False))                # safe default if key missing
    issue = str(verdict.get("issue", "")).strip()
    return {"verify_ok": ok, "verify_issue": issue, "history": state.history + [...]}

Design decision worth defending: if JSON parsing fails we default ok=False, so a garbled verifier triggers a revise rather than blindly accepting the answer — fail toward more checking.

The router is just an if-ladder returning a label the graph maps to a node:

def route_after_verify(state):
    if state.verify_ok:                   return "end"   # happy
    if state.iteration >= MAX_ITERATIONS: return "end"   # out of budget
    return "revise"                                      # loop

The eval's clever bit (eval_one / summarize): the agent returns its history, so we re-run the SQL it produced at each iteration against the DB and compare to gold. summarize then asks "what would the pass rate be if we'd stopped at iteration k?" with carry-forward. If iter-0 == last, the loop is decoration; if last is higher, it earns its keep — the whole point of Phase 5.
