1. SSH and port forwarding — what it means

SSH (Secure Shell) is an encrypted connection to a remote machine's terminal. You log into the cloud VM with it. But the VM's web UIs (Grafana, Langfuse, Prometheus) listen on localhost on the VM — not exposed to the internet. Port forwarding tunnels a port on your laptop through the SSH connection to a port on the VM, so your laptop browser can reach them safely.

ssh -L 3000:localhost:3000 \   # laptop:3000  ->  VM's localhost:3000 (Grafana)
    -L 9090:localhost:9090 \   # Prometheus
    -L 3001:localhost:3001 \   # Langfuse
    -L 8000:localhost:8000 \   # vLLM
    -L 8001:localhost:8001 \   # your agent
    user@vm-host

Each -L local:localhost:remote means "anything I open at http://localhost:local on my laptop, send through the tunnel to localhost:remote on the VM." With the session open, http://localhost:3000 in your browser hits Grafana on the VM. If a UI won't load, the forward is the first suspect.

2. What vLLM does

vLLM is a high-performance inference server for LLMs. You give it a model (Qwen3-30B) and it exposes an OpenAI-compatible HTTP API at :8000 — the same /v1/chat/completions shape OpenAI uses. Your agent sends prompts there and gets text back. Why vLLM instead of plain PyTorch:

  • Continuous batching — it merges many concurrent requests into one GPU batch, so 10 users don't wait in line; throughput scales.
  • PagedAttention / KV cache — it stores each request's attention state in GPU memory in pages, packing far more concurrent requests into 80GB. This is the capacity lever you tune.
  • /metrics endpoint — it publishes Prometheus metrics (latency, tokens/s, KV usage, queue depth) that your Grafana dashboard reads.
The OpenAI-compatible part is the whole trick. Because vLLM and OpenAI speak the same protocol, your agent code is identical whether it talks to a local vLLM or to OpenAI in the cloud. You only change a URL (see section 6).

3. Run on your Mac (off-GPU, no H100)

Everything except serving the 30B model and reporting latency works locally. The agent talks to OpenAI instead of vLLM. One-time setup is done; this is how to use it.

  1. Put OpenAI creds in .env (uncomment the block): 
    VLLM_BASE_URL=https://api.openai.com/v1
VLLM_MODEL=gpt-4o-mini
OPENAI_API_KEY=sk-...
  2. Start the o11y stack (optional locally — needed for Phase 4 Langfuse): 
    docker compose up -d
  3. Start the agent (uses the Mac dev venv with no vllm): 
    .venv-dev/bin/uvicorn agent.server:app --port 8001
  4. Smoke-test one question: 
    curl -s localhost:8001/answer -H 'content-type: application/json' \
  -d '{"question":"How many circuits are there?","db":"formula_1"}' | python3 -m json.tool
  5. Run the full eval against OpenAI (validates the harness; real numbers come from the H100): 
    .venv-dev/bin/python evals/run_eval.py --out results/eval_local.json

Already verified working on this Mac without any key:

server /health + /answer route registered✓
schema rendering, SQL execution on real BIRD DBs✓
eval comparison (gold==gold true, gold==wrong false)✓

Only the LLM call itself needs the key (or the GPU).

4. Run on the GPU VM (full system)

After forwarding the five ports (section 1), run on the VM:

  1. Setup once: uv sync · cp .env.example .env · uv run python scripts/load_data.py · docker compose up -d
  2. Phase 1 — serve vLLM: 
    bash scripts/start_vllm.sh        # add tuning flags here
# confirm: curl localhost:8000/v1/models
  3. Phase 2 — dashboard: open Grafana at localhost:3000 (admin/admin). The dashboard auto-loads from serving.json; fire requests and watch panels react.
  4. Phase 3 — agent: uv run uvicorn agent.server:app --host 0.0.0.0 --port 8001, then curl a question.
  5. Phase 4 — Langfuse: create a project at localhost:3001, paste public/secret keys into .env, restart the agent, fire 10 questions, inspect a trace.
  6. Phase 5 — baseline eval: uv run python evals/run_eval.py --out results/eval_baseline.json (watch Grafana — ~60 vLLM calls).
  7. Phase 6 — SLO: uv run python load_test/driver.py --rps 10 --duration 300. Diagnose from the dashboard, change one flag, re-run, log it. Then re-eval to results/eval_after_tuning.json.
  8. Phase 7 — REPORT.md and screenshots.

6. The backend seam — off-GPU to GPU, no code change

The agent reads three env vars (agent/graph.py). Switching backends is three lines:

Off-GPU (build/debug)On the H100 (real numbers)
VLLM_BASE_URLhttps://api.openai.com/v1http://localhost:8000/v1
VLLM_MODELgpt-4o-miniQwen/Qwen3-30B-A3B-Instruct-2507
OPENAI_API_KEYsk-...not-needed

Agent, eval, and load test are byte-identical in both worlds. Caveat: final prompt tuning and all reported numbers (eval pass rate, latency, SLO) must come from the real Qwen3 on the H100 — tokenization and behavior differ.

7. Observability at runtime — where the data goes

Two independent paths run during every request:

Serving x-rayAgent x-ray
ToolPrometheus + GrafanaLangfuse
SourcevLLM /metrics (scraped every 5s)LangGraph callback on each node
Answers"Is it slow, and where in the request lifecycle?""Why was this one request slow?"
Showslatency percentiles, tokens/s, KV cache, queueper-step waterfall: prompt, response, tokens, latency

In Phase 6 you read both together: Grafana tells you the GPU is KV-bound and queuing (the machine cause); Langfuse tells you verify fires on every question and doubles the LLM work (the logic cause). The eval then confirms whether a speed fix hurt quality. You can only see all three because you built all three instruments.
